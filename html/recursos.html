Cartas Pokemon (1º DAW/DAM)

SdA: Análisis, diseño y desarrollo de WebApp que genera 6 cartas Pokemon con la PokeAPI

Recursos

PokéAPI

API pública y gratuita con información de todos los Pokémon existentes. Es la fuente de datos en tiempo real de la webapp.

  • Web oficial: https://pokeapi.co
  • Endpoint de ejemplo (Pikachu): https://pokeapi.co/api/v2/pokemon/25

Probadla directamente en el navegador para identificar los datos que vais a necesitar (nombre, número de la Pokédex, tipos, sprites normales y shiny, estadísticas, habilidades).

Repositorio base del proyecto

No empezáis de cero. El profesor os entrega un repositorio base con la estructura del proyecto, algunos archivos preparados y la configuración mínima.

  • Repositorio: github.com/rsvives/pokecartas

Vuestro trabajo es forkearlo, completarlo y hacerlo vuestro. Todos los miembros del equipo deben añadirse como colaboradores.

Herramientas para diagramas UML

Una vez tengáis claro el modelo a lápiz/papel, lo pasáis a digital con cualquiera de estas herramientas:

  • draw.io (diagrams.net) — editor visual de diagramas en el navegador.
  • PlantUML — generación de diagramas a partir de código.
  • Mermaid — diagramas en Markdown (compatibles con GitHub).
  • StarUML — herramienta de modelado UML clásica.

Recordad: borrador a mano antes que a herramienta. Es más rápido para iterar y discutir.

Tablero Kanban (gestión del sprint)

Una sola fuente de verdad para las tareas: el tablero. Si una tarea no está en el tablero, no existe.

  • GitHub Projects — tablero integrado con el repositorio.
  • Trello — tablero Kanban sencillo y visual.
  • Versión física: pared del aula con post-its.

Tres columnas mínimas: To Do · Doing · Done. Una tarea solo está en "Done" si cumple la Definition of Done.

Git, GitHub y entorno de desarrollo

Vais a usar Git y GitHub en serio: ramas, commits descriptivos, pull requests internas y revisión entre compañeros.

  • Git — sistema de control de versiones.
  • GitHub — alojamiento del repositorio del equipo.
  • Visual Studio Code — IDE recomendado, con depurador integrado.

Convenciones del proyecto: una rama por sprint (al menos sprint-1 y sprint-2) y pull requests hacia main. Mejor diez commits pequeños y claros que uno enorme con todo.

Herramientas para la infografía

La infografía interactiva es el formato de presentación de la fase de análisis. Herramientas sugeridas:

  • Genially — infografías interactivas con tooltips, animaciones y elementos clicables.
  • Canva — diseño gráfico para infografías estáticas.
  • HTML estático con tooltips, si os animáis a maquetarlo a mano.

El formato de entrega es un enlace público a la infografía.

Herramientas para grabar el vídeo demo

Vídeo de 2-3 minutos máximo, con audio claro y demostración de la app funcionando. Mejor un vídeo claro y honesto que uno sobre-producido.

  • OBS Studio — multiplataforma, software libre.
  • Loom — grabación rápida desde el navegador.
  • QuickTime Player (macOS) — grabación de pantalla nativa.
  • Xbox Game Bar (Windows) — grabación nativa con Win + G.

Subidlo a Drive o YouTube (no listado) y enlazadlo en el README.

Truco para la grabación: ajustad temporalmente la probabilidad shiny al 100% para que aparezca seguro durante la demo. Acordaos de revertir el cambio antes de la entrega.
